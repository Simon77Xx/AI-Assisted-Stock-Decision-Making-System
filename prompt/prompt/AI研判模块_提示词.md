# 模块任务说明：AI 辅助行情研判服务（股海秘籍项目子模块）

## 背景与定位
本模块是"股海秘籍"项目中的一个独立子模块，职责单一：接收已经算好的结构化技术指标，调用大模型 API，返回结构化的市场研判结果。

整体项目架构：前端 React + Next.js，回测与数据分析部分用 Python/pandas 实现。本模块同样用 Python 实现，通过 FastAPI 封装成一个独立的 REST API 服务，由前端 React 直接调用（或经由 Next.js API Route 转发，具体看你们网络层怎么打通，本模块只需提供清晰的 HTTP 接口）。

请只实现这一个模块，不要涉及前端代码、不要涉及数据获取/回测逻辑（那些是其他模块的职责，本模块只负责"拿到指标 → 调用AI → 返回结构化结果"这一段）。

## 技术栈
- 语言：Python
- Web 框架：FastAPI
- 大模型调用：Anthropic Claude API（调用逻辑封装为独立函数，便于后续切换其他厂商）
- 数据校验：使用 Pydantic 定义请求/响应的数据模型

## 接口设计

### 请求（前端传入的结构化技术指标，不是原始K线）
设计一个 POST 接口，例如 `POST /api/ai-judgement`，请求体（用 Pydantic 模型定义）大致包含：
```json
{
  "stock_code": "600519",
  "stock_name": "贵州茅台",
  "current_price": 1680.5,
  "ma5": 1675.2,
  "ma20": 1660.8,
  "ma60": 1620.3,
  "ma_cross_status": "金叉",        // 金叉 / 死叉 / 无明显交叉
  "cross_date": "2026-06-20",       // 最近一次金叉/死叉发生日期，可为 null
  "trend_filter_status": "趋势市",   // 趋势市 / 震荡市
  "return_5d": 0.032,               // 最近5日涨跌幅
  "return_20d": 0.081,              // 最近20日涨跌幅
  "volume_ratio": 1.45,             // 当前成交量 / 近期均量
  "current_position": "持仓",        // 持仓 / 空仓
  "max_drawdown": -0.12             // 近期回测最大回撤，可为 null
}
```
具体字段可以根据实际回测模块已经算出的指标灵活调整，但核心原则不变：**只传入已经计算好的结构化技术事实，绝不传原始K线序列或让AI自己计算指标**。

### 响应（AI研判结果，必须结构化）
```json
{
  "judgment": "看多",              // 看多 / 看空 / 中性，仅这三个枚举值
  "confidence": "中",              // 高 / 中 / 低
  "reasons": [
    "MA5上穿MA20形成金叉，短期动能转强",
    "近20日涨幅8.1%，强于近5日表现，趋势延续概率较高",
    "当前处于趋势市，双均线信号有效性较高"
  ],
  "risk_note": "以上判断仅基于历史技术指标的辅助参考，不构成投资建议，市场存在不确定性，请结合自身判断谨慎决策。"
}
```

## 核心实现要求

1. **Prompt 设计**：将请求中的结构化指标拼装成清晰的文本，明确告诉模型"基于以下已计算好的技术指标给出研判，不要假设或编造未提供的数据"。在 system prompt 中明确要求模型只能返回合法 JSON，不要包含任何解释性前后缀文字。

2. **强制结构化输出**：调用 Claude API 时利用 tool use / structured output 能力（或在 prompt 中严格约束 JSON 格式 + 代码侧用 Pydantic 做二次校验），确保返回结果可被稳定解析为上述响应格式。`judgment` 字段必须是"看多/看空/中性"三选一，不允许模型自由发挥其他取值。

3. **异常处理（必须做）**：
   - API 调用失败（网络错误、超时、限流）：捕获异常，返回明确的错误状态码和提示信息，不能让整个服务崩溃
   - 返回内容不是合法 JSON 或字段缺失：捕获解析异常，返回兜底响应（如 `judgment: "中性", reasons: ["AI研判暂不可用，请稍后重试"]`），并记录日志方便排查
   - 设置合理的请求超时时间（如10-15秒），避免前端长时间卡住

4. **接口封装**：把"调用大模型生成研判"这一步封装成独立函数，例如 `get_ai_judgement(indicators: IndicatorInput) -> JudgementOutput`，函数内部再调用具体的 Claude API 客户端，方便后续替换模型供应商或做单元测试。

5. **API Key 管理**：通过环境变量读取（如 `ANTHROPIC_API_KEY`），不要硬编码在代码里，提供 `.env.example` 示例文件说明需要配置的环境变量。

6. **跨域配置**：由于是被 React/Next.js 前端调用，需要正确配置 CORS，允许前端域名访问（开发阶段可以先允许 localhost）。

## 明确不做的事
- 不做用户认证/鉴权（课程项目，本地单用户场景）
- 不做请求频率限制/缓存层（如果时间充裕可以加一个简单的内存缓存，避免短时间内对同一股票重复调用API，但这是可选加分项，不是必须项）
- AI 不输出具体买卖点位、目标价、仓位建议，只输出方向性判断（看多/看空/中性）+ 理由 + 风险提示，避免误导性的具体投资指令
- 不在本模块内做技术指标的计算（MA、涨跌幅等），假设这些都已经由调用方（其他模块）算好并通过请求传入

## 建议的开发与验证顺序
1. 先写一个独立的 Python 脚本（不经过 FastAPI），手动构造一组示例指标，验证"指标拼装 prompt → 调用 Claude API → 解析为结构化结果"这条链路能跑通，且输出格式稳定可靠
2. 链路验证无误后，再封装成 FastAPI 接口，补充请求/响应的 Pydantic 模型校验
3. 补充异常处理与兜底逻辑，用几个边界情况手动测试（如网络断开、模拟API返回异常格式等）
4. 最后做 CORS 配置，用前端实际发起请求联调

## 代码风格要求
- 接口层（FastAPI 路由）、业务逻辑层（指标转prompt、调用模型、解析结果）、模型客户端封装三者职责分离，不要全部写在一个路由函数里
- 关键的 prompt 内容和 JSON 格式约束需要有注释说明设计意图
- 返回给前端的错误信息要清晰可读，便于前端展示友好提示
